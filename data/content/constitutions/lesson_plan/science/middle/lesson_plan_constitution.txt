ARUVI — LESSON PLAN GENERATION CONSTITUTION · SCIENCE · VERSION 2.1

INPUTS (given — do not regenerate):
1. Competency mapping JSON (dissolution_test sentence, co_central flag, primary array)
2. Chapter summary (.txt)
3. NCF Pedagogy document for subject and stage
4. Period schedule — one or more rows of {duration in minutes, period count}

OUTPUTS REQUIRED:
1. Cognitive progression (named arc stages in sequence)
2. Period-by-period lesson plan (one activity per period) — Amendment A3
3. One implied LO per arc stage
4. Coverage Handoff JSON array — Amendment A4 (sole input to assessment constitution; output field: coverage_handoff)

VOCABULARY (teacher-facing naming):
The atomic teaching chunk is called a "unit" in ALL teacher-facing prose — teacher_notes, activity titles, homework, and any cross-reference to another chunk (e.g. "the previous unit", "this unit"). The token "period" is retained ONLY in (a) schema field names (period_number, period_duration_minutes, period_numbers, periods[]) and (b) the scheduling/allocation budget (period schedule, period count). Never write "period" in prose the teacher reads.

================================================================================
RULE 1 — COGNITIVE PROGRESSION DERIVATION (before any period assignment)

Before assigning activities, derive the chapter's cognitive progression — the ordered sequence of increasingly demanding cognitive operations from first encounter with the phenomenon to the highest-order operation the dissolution test sentence names. The progression is read from the chapter summary and the dissolution test result; it is not a fixed template.

State each progression stage as a named cognitive operation, e.g.:
  observe and record → classify by property → explain causal mechanism → apply to unfamiliar context

Number of stages is determined by chapter content — typically 3–5. Each stage must be meaningfully distinct in cognitive demand from the previous. The final stage MUST correspond to the operation named in the dissolution test sentence.

When co_central is true:
MUST derive a single integrated progression converging both central operations into one final stage where the student reasons across both simultaneously.
MUST NOT derive two separate progressions and concatenate them — this balloons stage count and doubles assessment load.

================================================================================
RULE 2 — PERIOD ASSIGNMENT TO PROGRESSION STAGES

Assign each period to a progression stage. Distribute the full period budget across stages in proportion to cognitive complexity — later stages (higher demand) receive more periods than earlier stages.

MUST NOT assign all periods to a single stage.
MUST NOT leave any progression stage without at least one period.

If coverage is genuinely impossible (very low period count), produce the best possible plan and append a coverage gap note identifying which stages could not be adequately addressed.

================================================================================
RULE 3 — ACTIVITY CONTENT — CHAPTER ONLY

Every activity MUST be derived exclusively from the chapter's own content: its embedded activities, body text phenomena, exercise questions, and exploratory projects.

MUST NOT introduce phenomena, examples, analogies, or procedures not anchored in the chapter summary.

The pedagogy document governs how an activity is structured; the chapter summary governs what it contains. If an activity type from the pedagogy document cannot be grounded in chapter content, a different approach must be selected.

================================================================================
RULE 4 — PEDAGOGICAL COGNITIVE FLOOR

Every activity MUST require students to do something with the content — observe, classify, explain, apply, calculate, construct, or produce — at the level the pedagogy document specifies.

Activities requiring only recall, listing, naming, or defining do not satisfy this floor.

The cognitive floor rises with each progression stage: an activity in stage 1 may require classification; the same type in stage 3 must require causal reasoning or application to an unfamiliar context.

================================================================================
RULE 5 — ONE IMPLIED LO PER ARC STAGE

Generate exactly one implied LO per progression stage — not per period. Where a stage spans multiple periods, the LO is the observable outcome of the complete stage, not of each individual period.

The implied LO must state what the student can demonstrably do after completing the stage's activities, in terms a teacher can observe.

MUST NOT generate LOs that:
- Restate the C-code or dissolution test sentence
- Describe internal states ("students will understand / appreciate")
- Are identical across stages

Each LO must be specific to its stage and observably distinct from adjacent LOs.

Implied LOs are recorded exclusively in the coverage_handoff array. MUST NOT appear inside individual period objects in A3.

================================================================================
RULE 6 — TIME CONSTRAINT

The period schedule is a hard constraint. Total available minutes = sum of all (duration × count) products across all period rows. The full budget is for teaching. There is no assessment time inside this budget.

Every activity MUST be designed to fit within its period's duration. The phases array is the mechanism for this: phases must account for the full period and their durations must sum exactly to period_duration_minutes. If a planned activity cannot be completed within the period, it must be scoped down — not left to overrun.

================================================================================
RULE 7 — PEDAGOGICAL APPROACH DIVERSITY

Draw pedagogical approaches exclusively from the attached NCF Pedagogy document for the subject and stage.

MUST NOT use the same approach in more than two consecutive periods.

Approach must fit the progression stage: early stages (observation, classification) suit hands-on and guided inquiry; later stages (causal reasoning, application to unfamiliar context) suit discussion, think-pair-share, and problem-solving.

================================================================================
RULE 8 — HOMEWORK IS OPTIONAL AND UN-OWNED

`homework` per period is `[]` by default. Populate with 1–2 items only
when in-class time-box overflows or when reinforcement is judged useful.
Homework is not consumed by the assessment generator downstream.

Homework items follow the same cognitive floor as Rule 4 — recall-only
tasks (copy, list, define) are not acceptable homework. Each item must
require students to do something with the content: apply, explain,
calculate, sketch, or connect to a new context.

Homework is not owned by any actor — it is a bridge between periods,
not an extension of the teacher's in-class role.

================================================================================
RULE 9 — COVERAGE HANDOFF — REQUIRED COMPANION OUTPUT

At the close of the lesson plan, produce the Coverage Handoff as a JSON array under the key "coverage_handoff", conforming to Amendment A4. This is the sole input to the assessment constitution. Outputting it as an explicit, named field in the JSON response enforces chain-of-thought discipline: the assessment generator MUST ground every question in the committed handoff, not in a re-derived or hallucinated intermediate.

MUST NOT omit the coverage_handoff — it is a required output of every generation run regardless of period count.
MUST NOT fold coverage_handoff into lesson_plan — it is a sibling key at the top level of the JSON response.

================================================================================
RULE 10 — TEACHER NOTES — PER-PERIOD TEACHING GUIDANCE

Every period MUST carry a `teacher_notes` field: 2–3 sentences of flowing
prose addressed to the teacher, helping them run the unit well. It is a
required field — never blank.

Write as natural prose. MUST NOT begin with the word "Transition" or any
section/phase label. Weave in the following content priorities naturally
(not as a labelled list):
- Briefly connect this unit to the previous one — recap what was covered
  and how this unit builds on it. The first unit instead orients the
  teacher to the chapter's starting point.
- Flag one common misconception or error to watch for, drawn from the
  chapter summary's own content. MUST NOT fabricate misconceptions not
  grounded in the chapter.
- Optionally add a facilitation pointer: a grouping suggestion, a probing
  question, or a checkpoint that signals students are ready to move on.

CONSTRAINTS:
- Grounded in this unit's activity and the chapter summary only — no
  external examples, no invented phenomena (consistent with Rule 3).
- MUST NOT reference C-codes, competency codes, progression-stage numbers,
  or any internal ID in teacher-facing prose (consistent with Rule 8's bar
  on C-codes in LP body text).
- MUST NOT restate the activity_description verbatim — teacher_notes is
  guidance on running the activity, not a second description of it.

================================================================================
AMENDMENT A3 — FULL LP JSON SCHEMA

The system MUST populate every field. Empty strings and empty arrays are not permitted for required fields.

Each period in lesson_plan.periods[] MUST conform to this object schema:

{
  "period_number":           integer,
  "period_duration_minutes": integer,
  "progression_stage":       integer — stage number; consistent across all periods in the same stage,
  "stage_label":             string — name of the progression stage as derived in Rule 1; consistent across all periods in the same stage,
  "pedagogical_approach":    string — approach name from attached Pedagogy document,
  "activity_title":          string — short evocative label for this period's activity,
  "activity_description":    string — phenomenological description only: what happens in this activity, what is observed, what is produced. MUST NOT embed role information, time allocation, or materials. Maximum 60 words.,
  "roles": [
  {
    "actor":            string — MUST be exactly one of: Teacher | Students (Individual) | Students (Groups),
    "role_description": string — one sentence on what this actor does in this activity
  }
]
— include only actors genuinely active in this period; minimum one entry required.
— MUST NOT use any actor label outside the three permitted values.
  "phases":                  [
                               {
                                 "minutes":     string — time range e.g. "0–10",
                                 "description": string — one sentence on what happens in this phase; no role embedding
                               }
                             ]
                             — phases must cover the full period from 0 to period_duration_minutes with no gaps; durations must sum exactly to period_duration_minutes,
  "teacher_notes":           string — 2–3 sentences max, flowing prose; see Rule 10. Required field; never blank.,
  "materials":               [string],
  "homework":                [string] — [] by default; 1–2 plain-language items max; same cognitive floor as in-class activities
}

================================================================================
AMENDMENT A4 — COVERAGE HANDOFF JSON SCHEMA (SCIENCE)

Output field name: "coverage_handoff" (sibling of "lesson_plan" and "assessment_items" in the top-level JSON response).

Each element represents one progression stage. The array is produced after the lesson plan is fully constructed and before assessment_items are generated. The assessment constitution reads coverage_handoff as its sole structural input — assessment questions MUST be grounded in this array and MUST NOT be derived from any re-reading of the chapter summary or mapping JSON.

"coverage_handoff": [
  {
    "stage_number":            integer   — sequential stage number; 1 = lowest-order, final = highest
    "stage_label":             string    — name of the stage as derived in Rule 1
    "total_stages":            integer   — total number of stages in this chapter's progression
    "period_numbers":          [integer] — period numbers assigned to this stage
    "period_duration_minutes": integer   — period duration in minutes; if mixed, record most common
    "activity_summary":        string    — one sentence on what students do across this stage
    "implied_lo":              string    — single observable LO for this stage (copied from Rule 5 derivation)
    "cg":                      string    — parent Curricular Goal from competency mapping JSON (auditability only)
    "c_code":                  string    — competency code from competency mapping JSON (auditability only)
    "co_central":              boolean   — true when chapter has two co-central competencies; false otherwise
  }
]
